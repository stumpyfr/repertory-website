Repertory.dev gives teams one place to publish the loops and skills their agents are allowed to conduct. The portable layer is the Agent Loops specification and OCI, not a single agent harness.

Why teams use it

Give agents the same trusted repertoire your best engineers use.

Onboarding

Bring a repo into agent work safely.

Pull a reviewed repertoire with the loops, skills, and guardrails a new agent needs before touching code.

Delivery

Run the same feature loop every time.

Ticket analysis, implementation, review, checks, and evidence can become a repeatable workflow instead of a one-off prompt.

Operations

Share release and incident routines.

Keep high-pressure work auditable with versioned loops that make escalation, outputs, and handoff explicit.

Shared primitives

A registry for scores, instruments, and approved repertoires.

Skills

Collections

Lock files

Repeatable execution

Resolved digests make agent workflows reproducible across machines, repositories, and time.

Auditable by default

Agents need a repertoire humans can inspect.

Digest pinning Pin the exact score and skill versions your team reviewed.
Signatures and attestations Attach provenance, validation, and evaluation evidence with OCI referrers.
Inspect before execution Review inputs, phases, dependencies, and escalation policy.
Traceable runs Keep run reports separate from the published workflow contract.

Why OCI

Not just a `SKILL.md` on GitHub.

A raw file over Git or HTTP is easy to fetch, but it is weak as a distribution contract. OCI gives loops and skills immutable digests, version tags, private registry auth, mirroring, collections, referrers for signatures and attestations, and lock files that let teams prove which workflow actually ran.

Harness-neutral

The workflow moves with the team.

Repertory.dev is designed around open Agent Loop documents and OCI-compatible registries. Teams can use the same score with different agents, runtimes, and local policies while preserving the contract, digest, and evidence trail.
